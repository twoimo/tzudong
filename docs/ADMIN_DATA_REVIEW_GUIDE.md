## 1. 개요

### 1.1 관리자 역할
- **데이터 검수**: 크롤링 파이프라인에서 생성된 맛집 데이터 검토
- **제보 관리**: 사용자가 제출한 신규/수정 제보 승인/거부
- **데이터 품질 유지**: 중복 방지, 정확한 좌표, 올바른 정보 확인

### 1.2 관리 대상 테이블
| 테이블 | 설명 |
|--------|------|
| `restaurants` | 맛집 데이터 (최종) |
| `restaurant_submissions` | 사용자 제보 메타데이터 |
| `restaurant_submission_items` | 제보 항목 (유튜브 링크별) |

### 1.3 상태 값 정의

#### restaurants.status
| 값 | 설명 |
|----|------|
| `pending` | 대기 (검수 전) |
| `approved` | 승인됨 (지도에 표시) |
| `deleted` | 삭제됨 (복원 가능) |

#### restaurant_submissions.status
| 값 | 설명 |
|----|------|
| `pending` | 검토 대기 |
| `approved` | 전체 승인 |
| `partially_approved` | 부분 승인 (일부 항목만) |
| `rejected` | 전체 거부 |

#### restaurant_submission_items.item_status
| 값 | 설명 |
|----|------|
| `pending` | 검토 대기 |
| `approved` | 승인됨 |
| `rejected` | 거부됨 |

## 3. 중복 검사 로직

### 3.1 알고리즘: Levenshtein Distance

```typescript
// 중복 판정 기준
const isDuplicate = 
    normalizedAddress === existingNormalizedAddress &&  // 정규화된 주소 일치
    nameSimilarity >= 0.85;                              // 이름 유사도 85% 이상
```

### 3.2 주소 정규화 규칙

```typescript
function normalizeAddress(address: string): string {
    return address
        // 층/호수 정보 제거 (같은 건물 다른 층 = 같은 주소)
        .replace(/지하\s*\d+\s*층/g, '')   // 지하1층, 지하 2층 등
        .replace(/지상\s*\d+\s*층/g, '')   // 지상1층, 지상 2층 등
        .replace(/\d+\s*층/g, '')          // 1층, 2층 등
        .replace(/\d+\s*호/g, '')          // 101호, 201호 등
        // 공백 및 특수문자 제거
        .replace(/\s+/g, '')
        .replace(/[^\w가-힣]/g, '')
        .toLowerCase();
}
```

### 3.3 중복 유형

| 유형 | 조건 | 처리 |
|------|------|------|
| **완전 중복** | 같은 주소 + 같은 이름 + 같은 YouTube 링크 | 자동 거부 |
| **병합 필요** | 같은 주소 + 같은 이름 + 다른 YouTube 링크 | 홈 상세패널에서 자동 병합 |
| **유사 중복** | 같은 주소 + 85% 이상 유사한 이름 | 관리자 확인 필요 |

### 3.4 병합 로직 (상세 패널)

`use-restaurants.tsx`의 `mergeRestaurants()` 함수:
- **조건**: 이름 100% 일치 OR (정규화된 주소 일치 + 이름 95% 이상 유사)
- **결과**: 같은 마커, 여러 유튜브 영상 표시

## 4. 관리자 데이터 검수 (AdminReviewPanel)

### 4.1 위치
- **파일**: `components/admin/AdminReviewPanel.tsx`
- **접근**: 홈 화면 → 관리자 메뉴 → "데이터 검수"

### 4.2 대상 데이터
- `restaurants` 테이블에서 `status = 'pending'`인 레코드
- 크롤링 파이프라인에서 자동 생성된 데이터

### 4.3 가능한 액션

#### 4.3.1 승인 (Approve)
```
┌─ 조건 ─────────────────────────────────────────┐
│ • 정보가 정확함                                 │
│ • 좌표가 올바름 (지오코딩 완료)                 │
│ • 중복이 아님                                   │
└─────────────────────────────────────────────────┘
        ↓
┌─ 결과 ─────────────────────────────────────────┐
│ • restaurants.status → 'approved'              │
│ • 지도에 표시됨                                 │
│ • 검색 결과에 노출                             │
└─────────────────────────────────────────────────┘
```

#### 4.3.2 수정 후 승인
```
┌─ 조건 ─────────────────────────────────────────┐
│ • 일부 정보가 틀림 (이름, 주소, 카테고리 등)   │
│ • 좌표 재검색 필요                              │
└─────────────────────────────────────────────────┘
        ↓
┌─ 과정 ─────────────────────────────────────────┐
│ 1. 수정 모달 열기                               │
│ 2. 정보 수정                                    │
│ 3. 지오코딩 재실행 (주소 변경 시)              │
│ 4. 저장                                         │
└─────────────────────────────────────────────────┘
        ↓
┌─ 결과 ─────────────────────────────────────────┐
│ • restaurants 레코드 업데이트                   │
│ • restaurants.status → 'approved'              │
└─────────────────────────────────────────────────┘
```

#### 4.3.3 삭제 (Soft Delete)
```
┌─ 조건 ─────────────────────────────────────────┐
│ • 잘못된 정보                                   │
│ • 중복 데이터                                   │
│ • 쯔양 방문 음식점이 아님                      │
└─────────────────────────────────────────────────┘
        ↓
┌─ 결과 ─────────────────────────────────────────┐
│ • restaurants.status → 'deleted'               │
│ • 지도에서 사라짐                               │
│ • 완전히 삭제되지 않음 (복원 가능)             │
└─────────────────────────────────────────────────┘
```

#### 4.3.4 삭제 후 복원 (Restore)
```
┌─ 조건 ─────────────────────────────────────────┐
│ • 잘못 삭제한 경우                              │
│ • 사용자 제보로 같은 맛집 재등록 요청 시        │
└─────────────────────────────────────────────────┘
        ↓
┌─ 과정 ─────────────────────────────────────────┐
│ 1. 삭제된 레코드 발견 다이얼로그 표시           │
│ 2. "복원" 버튼 클릭                             │
└─────────────────────────────────────────────────┘
        ↓
┌─ 결과 ─────────────────────────────────────────┐
│ • restaurants.status → 'approved'              │
│ • 기존 정보 유지 또는 새 정보로 업데이트        │
└─────────────────────────────────────────────────┘
```

## 6. YouTube 메타데이터 (필수)

### 6.1 메타데이터 개요

제보 승인 시 YouTube 메타데이터를 **반드시 가져와야** 합니다. 메타데이터는 영상의 정확성 검증과 광고 여부 판별에 사용됩니다.

### 6.2 메타데이터 항목

| 필드 | 설명 | 예시 |
|------|------|------|
| `title` | 유튜브 영상 제목 | "쯔양이 찾은 역삼동 맛집" |
| `published_at` | 영상 업로드 일시 | "2024-03-15T09:00:00Z" |
| `duration` | 영상 길이 | "PT15M30S" |
| `is_shorts` | 쇼츠 여부 | true / false |
| `is_ads` | 광고 영상 여부 | true / false |
| `what_ads` | 광고 업체명 (광고인 경우) | "XX치킨" / null |

### 6.3 메타데이터 가져오기 워크플로우

```
┌─────────────────────────────────────────────────────────────────┐
│ 제보 상세 화면                                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. 상태 확인                                                    │
│     └─ 초기 상태: "대기" (메타데이터 미수집)                    │
│                                                                  │
│  2. "메타데이터 가져오기" 버튼 클릭                              │
│     └─ YouTube Data API v3 호출                                 │
│     └─ OpenAI GPT-4o-mini로 광고 여부 분석                      │
│                                                                  │
│  3. 결과 표시                                                    │
│     ├─ 광고 아님: "광고 아님" 뱃지 (녹색)                       │
│     └─ 광고: "광고" 뱃지 + 업체명 (빨간색)                      │
│                                                                  │
│  4. 선택 박스                                                    │
│     └─ "이 항목 승인" 체크박스 (기본: 해제)                     │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 6.4 광고 분석 로직

```typescript
// OpenAI GPT-4o-mini 프롬프트
const prompt = `
당신은 한국어 유튜브 영상의 광고 여부를 판별하는 전문가입니다.
다음 유튜브 영상 제목에서 **협찬/광고/스폰서/PPL** 여부를 분석해주세요.

## 제목
{영상 제목}

## 분석 기준
- 협찬/스폰서/PPL/광고 관련 단어가 있는지 확인
- "리뷰만" 등 광고 아님을 명시한 경우 광고 아님
`;
```

### 6.5 메타데이터 저장

승인 시 `restaurants.youtube_meta` 컬럼에 저장:

```json
{
  "title": "쯔양의 역삼동 맛집 탐방",
  "published_at": "2024-03-15T09:00:00Z",
  "duration": "PT15M30S",
  "is_shorts": false,
  "is_ads": false,
  "what_ads": null
}
```

### 6.6 메타데이터 필수 조건

| 상황 | 메타데이터 필수 | 비고 |
|------|----------------|------|
| NEW 제보 승인 | ✅ 필수 | 새 레코드에 메타데이터 포함 |
| EDIT 제보 승인 | ⚠️ 권장 | 기존 메타데이터 사용 가능 |
| 여러 항목 제보 | ✅ 각각 필수 | 각 항목별로 메타데이터 수집 |

## 7. 지오코딩 필수 조건

### 7.1 지오코딩이 필요한 경우

| 상황 | 필수 여부 | 이유 |
|------|-----------|------|
| NEW 제보 승인 | ✅ 필수 | 새 레코드에 좌표 필요 |
| EDIT 제보 (주소 변경) | ✅ 필수 | 새 주소의 좌표 필요 |
| EDIT 제보 (주소 미변경) | ⚠️ 권장 | 기존 좌표 사용 가능 |
| 데이터 검수 승인 | ✅ 필수 | 크롤링 데이터 좌표 검증 |
| 데이터 검수 수정 | ✅ 필수 | 수정된 주소의 좌표 필요 |

### 7.2 지오코딩 없이 승인 불가


## 7.5 승인 필수 조건 요약

제보를 승인하기 위해 반드시 충족해야 하는 조건들입니다:

```
┌─────────────────────────────────────────────────────────────────┐
│                    승인 필수 조건 (UI 레벨)                       │
├─────────────────────────────────────────────────────────────────┤
│ 1. 지오코딩 완료 (geocodingDone)                                 │
│    └─ lat, lng, road_address 모두 존재                         │
│                                                                  │
│ 2. 맛집명 입력 (hasName)                                         │
│    └─ editableData.name.trim() 존재                            │
│                                                                  │
│ 3. 최소 1개 항목 선택 (hasSelectedItem)                          │
│    └─ itemDecisions 중 approved=true인 항목 존재               │
│                                                                  │
│ 4. 선택된 모든 항목 메타데이터 완료 (selectedItemsMetaFetched)   │
│    └─ 승인 체크된 모든 항목의 metaFetched=true                 │
└─────────────────────────────────────────────────────────────────┘

canApprove = geocodingDone && hasName && hasSelectedItem && selectedItemsMetaFetched
```

### 7.6 중복 검사 (승인 실행 시)

승인 버튼 클릭 후 **mutation 실행 시** 추가로 중복 검사가 수행됩니다:

```
┌─────────────────────────────────────────────────────────────────┐
│                    중복 검사 (서버 레벨)                          │
├─────────────────────────────────────────────────────────────────┤
│ 조건: forceApprove === false (강제 승인 미체크)                  │
│                                                                  │
│ 검사 내용:                                                       │
│   1. 정규화된 지번주소 일치 검사                                 │
│      └─ 공백 제거, 층/호수 제거 (지하n층, n호 등)              │
│                                                                  │
│   2. 이름 유사도 검사 (Levenshtein Distance)                    │
│      └─ 유사도 85% 이상 → 중복으로 판정                        │
│                                                                  │
│ 중복 발견 시:                                                    │
│   - 에러 throw → 승인 실패                                      │
│   - 기존 맛집 정보 표시                                          │
│   - "강제 승인하려면 '중복 무시' 옵션을 체크하세요" 안내        │
│                                                                  │
│ 우회 방법:                                                       │
│   - forceApprove = true 체크 → 중복 검사 스킵                   │
│   ⚠️ 남용 시 데이터 품질 저하                                   │
└─────────────────────────────────────────────────────────────────┘
```

```typescript
// evaluations/page.tsx - approveSubmissionMutation
if (!forceApprove) {
  const duplicateResult = await checkRestaurantDuplicate(
    editableData.name,
    approvalData.road_address || approvalData.jibun_address || ''
  );

  if (duplicateResult.isDuplicate && duplicateResult.matchedRestaurant) {
    throw new Error(
      `중복 맛집이 발견되었습니다: "${duplicateResult.matchedRestaurant.name}"
       유사도: ${duplicateResult.similarityScore.toFixed(0)}%`
    );
  }
}
```

### 7.7 승인 버튼 활성화 조건 (코드)

```typescript
// SubmissionListView.tsx
const canApprove = useMemo(() => {
    if (!selectedSubmission) return false;
    
    // 1. 지오코딩 완료 여부
    const geocodingDone = !!approvalData.lat && !!approvalData.lng && !!approvalData.road_address;
    
    // 2. 최소 1개 항목 선택
    const hasSelectedItem = Object.values(itemDecisions).some(d => d.approved);
    
    // 3. 선택된 모든 항목 메타데이터 완료
    const selectedItemsMetaFetched = Object.entries(itemDecisions)
        .filter(([, d]) => d.approved)
        .every(([, d]) => d.metaFetched);
    
    // 4. 맛집명 입력
    const hasName = !!editableData.name.trim();
    
    return geocodingDone && hasSelectedItem && selectedItemsMetaFetched && hasName;
}, [approvalData, selectedSubmission, itemDecisions, editableData.name]);

// ⚠️ 중복 검사(이름 유사도 + 주소)는 버튼 클릭 후 mutation에서 수행됨
```

## 10. FAQ

### Q1: NEW와 EDIT의 차이점은?

| 구분 | NEW (신규) | EDIT (수정) |
|------|-----------|-------------|
| 목적 | 새 맛집 등록 | 기존 맛집 정보 수정 |
| target_restaurant_id | NULL (item 레벨) | 수정 대상 레코드 ID (item 레벨) |
| 승인 결과 | INSERT | UPDATE |
| 중복 검사 | 전체 검사 | 자기 자신 제외 |

> **⚠️ 참고**: `target_restaurant_id`는 `restaurant_submissions` 테이블이 아닌 `restaurant_submission_items` 테이블에서 각 항목별로 관리됩니다. 한 제보에 여러 유튜브 링크가 있는 경우, 각 링크별로 독립적인 기존 레코드(`restaurants.id`)와 매칭됩니다.

### Q2: unique_id는 어떻게 생성되나요?

```typescript
// SHA-256 해시
const uniqueIdString = youtubeLink + name + tzuyangReview;
const hash = await crypto.subtle.digest('SHA-256', encode(uniqueIdString));
const uniqueId = Array.from(new Uint8Array(hash))
    .map(b => b.toString(16).padStart(2, '0'))
    .join('');
```

- **구성 요소**: youtube_link + name + tzuyang_review (구분자 없음)
- **알고리즘**: SHA-256
- **용도**: 레코드 고유 식별, 중복 방지

### Q3: 병합은 언제 발생하나요?

- **장소**: 홈 화면 상세 패널
- **조건**: 
  - 이름 100% 일치, 또는
  - 정규화된 주소 일치 + 이름 95% 이상 유사
- **결과**: 하나의 마커, 여러 유튜브 영상 표시

### Q4: 강제 승인은 언제 사용하나요?

- 중복 검사에서 유사한 맛집이 발견되었지만, **실제로는 다른 맛집**일 때
- 예: "역삼동 맛집" vs "역삼동 맛집 본점" (같은 주소, 다른 가게)
- ⚠️ 남용 시 데이터 품질 저하

### Q5: 삭제된 레코드는 복원 가능한가요?

- **가능합니다.** `status = 'deleted'`인 레코드는 DB에 남아있음
- **복원 시점**:
  - 관리자가 직접 복원
  - 사용자 제보 승인 시 자동 감지
- **완전 삭제**: 현재 지원하지 않음 (데이터 보존 정책)
