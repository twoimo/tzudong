# ⚖️ 법적 리스크 분석 보고서 (2025 통합본)

본 보고서는 `Tzudong Map` 프로젝트의 코드베이스, 데이터 파이프라인(`backend`), 및 프론트엔드(`apps/web`) 로직을 정밀 분석하여 도출된 법적 리스크와 대응 방안을 통합 정리한 문서입니다.

(기존 `docs/LEGAL_RISK_ANALYSIS.md`와 `docs/LEGAL_RISK_REPORT_2025.md` 내용을 통합)

## 🚨 1. 퍼블리시티권 및 부정경쟁방지법 (Critical)

**현황 및 문제점:**
- **서비스명 및 아이덴티티**: '쯔동여지도'라는 명칭과 로고, 핵심 콘텐츠가 인플루언서 '쯔양'의 성명, 초상, 브랜드 식별력에 전적으로 의존하고 있습니다.
- **라이선스 부재**: 코드베이스 전반에서 공식 제휴 계약이나 라이선스 사용 권한을 증명할 수 있는 로직이나 문서가 발견되지 않았습니다.

**법적 리스크:**
- **퍼블리시티권 침해**: 당사자의 동의 없이 성명이나 초상을 영리/비영리 목적으로 이용하는 것은 명백한 권리 침해입니다.
- **부정경쟁행위**: 타인의 상당한 투자나 노력으로 만들어진 성과(쯔양의 브랜드, 맛집 리스트)를 무단으로 사용하여 경제적 이익을 침해하는 행위(부정경쟁방지법 제2조 1호 파목)에 해당할 수 있습니다.
- **가능성**: 소속사로부터의 서비스 중단 가처분 신청, 도메인 압수, 손해배상 청구 소송 가능성이 매우 높습니다 (Risk Level: 100%).

**대응 방안:**
1.  **공식 제휴**: 소속사와 공식 라이선스 계약을 체결하십시오.
2.  **비영리 팬 사이트 전환**: 제휴가 불가능하다면, "본 서비스는 쯔양님과 무관한 비공식 팬 사이트입니다"라는 문구(Disclaimer)를 서비스 전면에 명시하고, 어떠한 수익 모델(광고, 후원 등)도 도입하지 마십시오.

## 🔐 2. 개인정보보호법 및 정보통신망법 (Critical)

**관련 파일:**
- `apps/web/components/reviews/ReviewModal.tsx`
- `backend/geminiCLI-ocr-receipts/ocr-receipts.js`

**현황 및 문제점:**
- **영수증 OCR 처리**: 사용자가 업로드한 영수증(`verification_photo`)을 `review-photos` 버킷에 저장하고, 백엔드 스크립트에서 **`getPublicUrl`**을 통해 접근하여 Gemini API로 전송합니다.
- **Public URL 사용**: `getPublicUrl`을 사용한다는 것은 해당 이미지가 추측 가능한 퍼블릭 URL을 통해 누구에게나 접근 가능하다는 의미입니다.
- **제3자 제공**: OCR 분석을 위해 Google(Gemini API)에 이미지 데이터를 전송합니다.

**법적 리스크:**
- **개인정보 유출**: 영수증에는 카드번호(일부), 승인번호, 정확한 결제 시간, 카드사 정보 등 개인 식별 가능 정보(PII)와 사적인 동선 정보가 포함됩니다. 이를 노출하는 것은 심각한 보안 사고입니다.
- **제3자 제공 의무 위반**: 사용자에게 "OCR 분석을 위해 구글에 이미지를 제공함"을 고지하고 "국외 이전 동의"를 받지 않았다면 개인정보보호법 위반입니다.

**대응 방안:**
1.  **스토리지 보안 (즉시)**: Supabase `review-photos` 버킷을 **Private**으로 전환하십시오.
2.  **Signed URL 적용 (단기)**: 서버 사이드에서 유효기간이 짧은 Signed URL을 생성하여 OCR 서버에 전달하도록 수정하십시오.
3.  **프론트엔드 보안**: 다른 사용자가 타인의 영수증 사진을 절대 볼 수 없도록 UI/UX를 점검하십시오.

## 📹 3. YouTube API 이용 약관 (High)

**관련 파일:**
- `apps/web/hooks/use-restaurants.tsx`
- `backend/geminiCLI-restaurant-crawling/scripts/api-youtube-meta.py`
- `backend/geminiCLI-restaurant-crawling/scripts/crawling-pipeline.py`

**현황 및 문제점:**
- **데이터 영구 저장**: `api-youtube-meta.py`는 Video ID뿐만 아니라 제목, 설명, 썸네일 등 메타데이터를 `restaurants` 테이블의 `youtube_meta` 컬럼에 영구 저장합니다.
- **외부 스크래핑**: `crawling-pipeline.py`는 `maestra.ai` 등 타사 자막 사이트를 스크래핑(Puppeteer)하는 로직을 포함하고 있습니다.

**법적 리스크:**
- **YouTube Developer Policies (III.E.4)**: 비디오 ID를 제외한 API 데이터를 30일 이상 저장(Caching)하는 것을 금지합니다. 위반 시 API 키 영구 정지 사유가 됩니다.
- **서비스 약관 위반**: 타사 서비스의 데이터를 무단 스크래핑하는 것은 해당 서비스의 약관 위반이며 IP 차단 대상입니다.

**대응 방안:**
1.  **데이터 갱신(Refresh) (중기)**: 메타데이터 저장 시 `last_updated` 컬럼을 두고, 30일마다 API를 호출하여 데이터를 갱신하거나 삭제하는 스케줄러(Cron Job)를 구현하십시오.
2.  **DB 최소화**: 가능하다면 DB에는 Video ID만 저장하고, 메타데이터는 클라이언트에서 실시간으로 가져오도록 변경하십시오.

## 🗺️ 4. 데이터베이스권 및 기타 (Medium)

**관련 파일:**
- `apps/web/hooks/use-restaurants.tsx` (병합 로직)
- `backend/geminiCLI-ocr-receipts/ocr-receipts.js` (매칭 로직)

**현황 및 문제점:**
- **초기 데이터 출처**: 초기 `restaurants` 테이블이 네이버/카카오 지도 등을 대량 크롤링하여 구축되었을 가능성이 있는 구조(매칭 로직 등)가 보입니다.
- **UGC 저작권**: `ReviewModal`을 통해 사용자가 업로드하는 음식 사진(`food_photos`)이 본인이 찍은 것이 아닐 수 있습니다.

**법적 리스크:**
- **부정경쟁방지법/데이터베이스권**: 타 지도 서비스의 POI 데이터를 무단 복제하여 경쟁 서비스를 제공하는 것은 불법입니다.
- **저작권 방조 책임**: 플랫폼은 사용자 침해 게시물에 대한 관리 책임이 있습니다.
- **지도 API 약관**: `NaverMapView.tsx` 사용 시 네이버 로고를 CSS로 가리거나 덮으면 약관 위반입니다.

**대응 방안:**
1.  **출처 검증**: 맛집 데이터가 순수하게 '사용자 제보'와 '공식 API 검색'으로만 축적되도록 프로세스를 확립하십시오. 대량 크롤링 스크립트가 있다면 폐기하십시오.
2.  **이용약관(Terms of Use)**: "사용자가 업로드한 콘텐츠의 저작권 책임은 사용자에게 있음"을 명시하고, 회사에 홍보 목적 라이선스를 허락하는 조항을 넣으십시오.
3.  **UI 점검**: 네이버/구글 지도 로고가 항상 보이도록 여백을 확보하십시오.
