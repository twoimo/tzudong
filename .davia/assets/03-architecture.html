시스템 아키텍처

쯔동여지도가 어떻게 구성되어 있는지 전체적인 그림을 설명합니다.

아키텍처란?

아키텍처는 건물의 설계도처럼, 소프트웨어가 어떤 부품들로 이루어져 있고, 그 부품들이 서로 어떻게 연결되어 있는지를 보여주는 것입니다.

쯔동여지도는 크게 3개의 부품으로 이루어져 있어요:

전체 시스템 아키텍처

아래 다이어그램에서 각 컴포넌트가 어떻게 연결되어 있는지 확인하세요:

다이어그램 상세 설명

Frontend (웹 프론트엔드)

사용자가 직접 보고 상호작용하는 웹 애플리케이션 계층입니다.

  • Map - Naver/Google Maps API를 활용한 맛집 위치 시각화. 클러스터링/마커 클릭 이벤트 처리
  • Search - 맛집명, 주소, 지역 기반 검색. Supabase Full-Text Search 활용
  • Detail - 맛집 상세 정보 표시. 유튜브 영상 링크, 리뷰, 좌표 정보 포함

Pipeline (데이터 수집 시스템)

AI를 활용해 유튜브 영상에서 맛집 정보를 자동으로 추출하는 백엔드 시스템입니다.

  • Transcript - GitHub Actions + Puppeteer로 자막 수집 (maestra.ai → tubetranscript.com 폴백)
  • Crawl - Gemini CLI로 자막에서 맛집 정보(name, address, menu) 추출
  • Eval - 규칙 기반 검증 + LLM as a Judge로 데이터 품질 평가
  • Transform - Naver Geocoding API로 주소→좌표 변환, 중복 제거 처리

Supabase (데이터베이스 & 백엔드)

PostgreSQL 기반의 BaaS(Backend as a Service)로, 데이터 저장과 인증을 담당합니다.

  • restaurants - 맛집 정보 테이블 (이름, 주소, 좌표, 메뉴, 유튜브 링크 등)
  • reviews - 사용자 리뷰 테이블 (rating, comment, user_id, restaurant_id)
  • Auth - 사용자 인증 시스템 (Google/Kakao OAuth 지원)

External (외부 서비스 연동)

프로젝트에서 사용하는 외부 API 및 서비스들입니다.

  • YouTube API - 영상 메타데이터 및 자막 추출용
  • Gemini AI - Google의 LLM으로 맛집 정보 추출 및 평가 수행
  • Naver Maps - 지도 표시 및 Geocoding (address → lat/lng)
  • Google Maps - 대체 지도 옵션 및 Place ID 조회

1. 프론트엔드 (사용자가 보는 화면)

역할: 사용자가 웹 브라우저에서 보는 모든 화면을 담당해요.

사용 기술

  • Next.js 16 - 웹사이트를 만드는 프레임워크 (React 기반)

  • Turbopack - 개발할 때 빠르게 변경사항을 반영해주는 도구

  • TypeScript - 자바스크립트에 타입을 추가해서 버그를 줄여줌

  • Tailwind CSS - 스타일링을 빠르게 할 수 있는 CSS 도구

  • Radix UI - 버튼, 모달 같은 UI 컴포넌트 라이브러리

상태 관리 (데이터를 어떻게 관리하나요?)

  • React Query - 서버에서 가져온 데이터 관리 (맛집 목록, 리뷰 등)

  • Zustand - 화면에서만 쓰는 임시 데이터 관리 (필터 상태, 모달 열림/닫힘 등)

지도 기능

  • 네이버 지도 - 국내 맛집 표시용 (한국에서 더 정확함)

  • 구글 지도 - 해외 맛집 표시용

프론트엔드가 하는 일

  1. 사용자에게 지도 화면을 보여줌

  2. 맛집을 클릭하면 상세 정보를 표시

  3. 검색과 필터 기능 제공

  4. 사용자의 리뷰 작성을 처리

  5. 로그인/로그아웃 처리

2. 데이터베이스 (데이터 저장소)

역할: 모든 데이터를 안전하게 저장하고, 필요할 때 꺼내주는 역할이에요.

사용 기술

  • Supabase - 데이터베이스 서비스 (PostgreSQL 기반)

  • PostgreSQL - 실제 데이터가 저장되는 데이터베이스 엔진

왜 Supabase를 선택했나요?

  • 무료 시작 - 작은 프로젝트는 무료로 사용 가능

  • 인증 내장 - 이메일/비밀번호 로그인 기능이 이미 만들어져 있음

  • 실시간 기능 - 데이터가 바뀌면 자동으로 화면에 반영 가능

  • API 자동 생성 - 테이블 만들면 API가 자동으로 생성됨

주요 기능

  • RLS (Row Level Security) - 데이터 접근 권한 관리. 예를 들어, 일반 사용자는 승인된 맛집만 볼 수 있고, 관리자는 모든 맛집을 볼 수 있음

  • Auth - 이메일/비밀번호로 로그인 기능

  • Realtime - 새 리뷰가 달리면 실시간으로 화면에 표시

데이터베이스에 저장되는 것들

  • 맛집 정보 - 이름, 주소, 좌표, 쿠양 리뷰 등

  • 유튜브 영상 정보 - 영상 제목, 링크, 썸네일 등

  • 사용자 정보 - 프로필, 역할(일반/관리자) 등

  • 리뷰 정보 - 리뷰 내용, 사진, 별점 등

3. AI 파이프라인 (데이터 수집 시스템)

역할: 유튜브 영상을 분석해서 맛집 정보를 자동으로 수집하고 검증하는 역할이에요.

사용 기술

  • Gemini CLI - Google의 AI(Gemini)를 명령줄에서 사용하는 도구

  • Python - 파이프라인 스크립트 작성 언어

  • YouTube Transcript API - 유튜브 영상의 자막을 가져오는 도구

  • Naver Geocoding API - 주소를 지도 좌표로 변환하는 도구

파이프라인이 하는 일

  1. 영상 수집 - 쯔양 채널의 영상 목록을 가져옴

  2. 자막 추출 - 각 영상의 자막(말하는 내용)을 추출

  3. AI 분석 - Gemini AI가 자막을 읽고 맛집 정보를 찾아냄

  4. 좌표 변환 - 주소를 위도/경도로 변환

  5. 검증 - RULE과 LAAJ 평가로 정보 검증

  6. 저장 - 검증된 정보를 데이터베이스에 저장

두 가지 파이프라인

원래는 Perplexity AI를 썼지만, 더 안정적인 Gemini CLI로 바꿨어요:

  • GeminiCLI (현재 사용) - Google 공식 도구라서 안정적이고 합법적

  • Perplexity (대체용) - 웹 검색 기능이 있지만 브라우저 자동화라 불안정

데이터가 흘러가는 과정

전체 흐름을 순서대로 설명하면:

1단계: 데이터 수집 (AI 파이프라인 데이터베이스)

유튜브 영상  자막 추출  AI 분석  좌표 변환  검증  DB 저장

2단계: 데이터 조회 (프론트엔드 데이터베이스)

사용자가 지도를 봄  프론트엔드가 DB에 요청  DB가 맛집 목록 반환  지도에 표시

3단계: 사용자 활동 (프론트엔드 데이터베이스)

사용자가 리뷰 작성  프론트엔드가 DB에 저장  다른 사용자도 리뷰를 볼 수 있음

보안은 어떻게 관리하나요?

인증 (Authentication)

사용자가 누구인지 확인하는 과정:

  • 이메일/비밀번호 인증 - Supabase Auth를 통한 회원가입 및 로그인

  • 닉네임 설정 - 회원가입 시 닉네임 필수 입력

  • 세션 관리 - Supabase Auth가 자동으로 세션 유지

권한 (Authorization)

누가 무엇을 할 수 있는지 정하는 것:

  • 일반 사용자 (user) - 승인된 맛집 보기, 리뷰 작성, 제보하기

  • 관리자 (admin) - 모든 맛집 보기, 제보 승인/거부, 데이터 수정

RLS (Row Level Security)

데이터베이스 수준에서 접근 제어:

  • 일반 사용자는 status = 'approved'인 맛집만 조회 가능

  • 리뷰는 본인 것만 수정/삭제 가능

  • 관리자는 모든 데이터에 접근 가능

API 키 관리

  • 모든 API 키는 환경 변수로 관리 (코드에 직접 넣지 않음)

  • anon key - 일반 사용자용 (제한된 권한)

  • service_role key - 백엔드용 (관리자 권한, 절대 공개하면 안 됨)
