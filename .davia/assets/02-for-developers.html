개발자 문서

쯔동여지도 프로젝트의 기술 문서입니다. 이 문서는 프로젝트에 참여하거나 코드를 이해하고 싶은 개발자를 위해 작성되었습니다.

GeminiCLI 파이프라인 상세 흐름

데이터 수집부터 DB 저장까지의 전체 과정을 다이어그램으로 확인하세요:

이 프로젝트 이해하기

쯔동여지도는 크게 3가지 부분으로 구성됩니다:

  1. 프론트엔드 (웹사이트) - 사용자가 보는 화면. Next.js로 만들었어요
  2. 백엔드 파이프라인 - AI가 맛집 정보를 수집하는 시스템. Python과 Gemini CLI 사용
  3. 데이터베이스 - 수집된 데이터를 저장하는 곳. Supabase(PostgreSQL) 사용

기술 스택 상세

각 기술이 하는 역할:

  • Next.js 16 - React 기반 웹 프레임워크. 서버 사이드 렌더링(SSR)과 정적 생성(SSG) 지원
  • TypeScript - 자바스크립트에 타입을 추가해서 버그를 미리 잡아줌
  • Tailwind CSS - 클래스 기반으로 빠르게 스타일링 가능
  • Radix UI - 접근성이 좋은 UI 컴포넌트 라이브러리
  • React Query - 서버 데이터 fetching과 캐싱을 쉽게 해줌
  • Zustand - 가벼운 상태 관리 라이브러리
  • Supabase - PostgreSQL 데이터베이스 + 인증 + 실시간 기능 제공
  • Gemini CLI - Google의 AI 모델을 명령줄에서 사용할 수 있게 해주는 도구

데이터 파이프라인 이해하기

AI가 맛집 정보를 수집하고 DB에 저장하는 과정입니다:

각 단계 자세히 설명:

1단계: 크롤링 (정보 수집)

  1. 유튜브 영상 URL 목록을 준비해요
  2. 각 영상의 자막(대화 내용)을 가져와요
  3. AI(Gemini)가 자막을 읽고 맛집 정보를 추출해요
  4. 추출된 주소로 지도 좌표(위도/경도)를 찾아요

2단계: 평가 (검증)

  1. RULE 평가 - 규칙으로 빠르게 검증 (주소가 실제로 있는지 등)
  2. LAAJ 평가 - AI가 다시 한번 꼼꼼히 검토

3단계: 에러 처리

실패한 항목은 최대 5번까지 자동으로 재시도해요

4단계: 데이터 변환

DB에 넣기 좋은 형태로 데이터를 정리해요

5단계: DB 저장

중복 확인 후 Supabase 데이터베이스에 저장해요

파이프라인 실행 명령어:

# 전체 파이프라인 한 번에 실행
cd backend/geminiCLI-restaurant-crawling

# 평가만 따로 실행
cd backend/geminiCLI-restaurant-evaluation  
python scripts/evaluation-pipeline.py

# DB에 데이터 넣기
npx tsx scripts/insert_to_supabase.ts

RULE vs LAAJ 평가 시스템

AI가 수집한 정보가 맞는지 두 가지 방법으로 검증합니다:

RULE 평가 (빠른 1차 검증)

정해진 규칙으로 빠르게 걸러내요:

  • 주소 검증 - 네이버 Geocoding으로 실제 주소인지 확인
  • 영상 출현 여부 - 자막에 맛집 이름이 언급되었는지
  • 카테고리 매칭 - 음식점 카테고리가 올바른지

LAAJ 평가 (정밀한 2차 검증)

LAAJ = LLM As A Judge (AI를 심판으로 사용)

AI가 다시 한번 꼼꼼히 검토해서 점수를 매겨요:

  • flavor_rating (맛 점수) - 쯔양님 반응 기반 1-10점
  • quantity_rating (양 점수) - 음식 양 기반 1-10점
  • actual_visit_confirmed - 실제 방문했는지 true/false
  • is_in_video - 영상에 실제로 나오는지 true/false

API 사용하기

Supabase를 통해 데이터를 조회하고 수정할 수 있어요:

맛집 목록 가져오기

// React 컴포넌트에서 사용 예시
import { useQuery } from '@tanstack/react-query'
import { supabase } from '@/lib/supabase'

function RestaurantList() {
  const { data, isLoading, error } = useQuery({
    queryKey: ['restaurants'],
    queryFn: async () => {
      // Supabase에서 승인된 맛집만 가져오기
      const { data, error } = await supabase
        .from('restaurants')
        .select('*')
        .eq('status', 'approved')  // 승인된 것만
        .order('created_at', { ascending: false })  // 최신순
        .limit(20)  // 20개만
      
      if (error) throw error
      return data
    }
  })

  if (isLoading) return <div>로딩 중...</div>
  if (error) return <div>에러 발생!</div>

  return (
    <ul>
      {data.map(restaurant => (
        <li key={restaurant.id}>{restaurant.name}</li>
      ))}
    </ul>
  )
}

주요 API 엔드포인트

  • GET /rest/v1/restaurants - 맛집 목록
  • GET /rest/v1/restaurants?id=eq.{id} - 특정 맛집 상세
  • GET /rest/v1/youtube_videos - 유튜브 영상 목록
  • GET /rest/v1/reviews - 리뷰 목록
  • POST /rest/v1/reviews - 리뷰 작성 (로그인 필요)

성능 최적화 팁

프론트엔드 최적화

  • 코드 스플리팅 - 지도 컴포넌트는 dynamic import로 필요할 때만 로드
  • 이미지 최적화 - next/image 사용, AVIF/WebP 포맷
  • React Query 캐싱 - staleTime 설정으로 불필요한 요청 방지

적용 결과

  • Lighthouse 점수: 24점 85점 이상
  • 페이지 로딩 시간: 3.5초 2초

자주 발생하는 문제와 해결법

문제: Gemini API 요청 한도 초과

증상: 429 Too Many Requests 에러

해결: 요청 간격을 늘리거나, 배치 크기를 줄이세요

문제: 좌표 변환 실패

증상: latitude/longitude가 null

해결: 주소 형식이 네이버 Geocoding에서 인식 안 될 수 있어요. 주소를 정규화하거나 수동으로 좌표를 입력하세요

문제: Supabase RLS 오류

증상: row-level security policy violation

해결: 백엔드에서는 service_role 키를 사용해야 해요. anon 키로는 제한된 작업만 가능합니다

더 많은 문제 해결 방법은 트러블슈팅 가이드를 참조하세요.
